# 棠溪 · 分层分析策略 v1.0
# 叠加在 v9.7 模板之上

## 核心原则

高周期收线时分析一次 → 缓存结果 → 低周期复用直到下根K收线。
避免每个周期都全量跑分析。

## 周期分层

```
4h ── 定方向·辨市态 ── 每4h收线时分析 ── 缓存：方向/市态/关键位
1h ── 结构·价值区  ── 每1h收线时分析 ── 继承4h缓存做过滤
15m ─ 执行·关键位  ── 行情守望10s实时  ── 继承1h结构做触发判定
5m  ─ 入场触发    ── 15m触发后切换    ── 确认条件满足后入场
```

## 更新规则

| 事件 | 触发动作 |
|------|---------|
| 4h K收线 | 重跑完整分析 → 更新4h缓存 + 递归刷新1h |
| 1h K收线 | 重跑1h结构分析 → 继承4h缓存做过滤 |
| 15m关键位触发 | 行情守望推送 → 出短执行卡 |
| 5m确认条件满足 | 出执行确认 → 入场 |
| 4h缓存过期（>4h未更新） | 强制重跑全部分析 |
| 1h缓存过期（>1h未更新） | 强制重跑1h分析 |

## 分析卡出卡规则

| 场景 | 卡片类型 | 内容 |
|------|---------|------|
| 4h收线 | 长卡 | 全5段·带截图 |
| 1h收线+方向变化 | 中卡 | 头部+博弈+操作·带截图 |
| 15m关键位触发 | 短卡 | 触发位+动作+风控 |
| 5m确认入场 | 执行卡 | 方向/入场/止损/止盈/仓位 |

## 实施方式

分析缓存写入 `data/analysis_cache.json`。
行情守望的 `maybe_request_refresh()` 已自动写 `structure_refresh_requests.jsonl`。
Agent 读取刷新请求 + 分析缓存判断是否需要出卡。
